测试用例设计

测试用例包含内容

软件测试活动中，最重要的内容就是测试用例，我们首先看一下测试用例包含什么内容，各个企业要求点可能有所差异，但核心部分大致相同。
1.功能模块(测试项) ： 对应的产品功能模块是哪部分
2.检查项(测试标题) ： 检查的项目是哪个类别，比如说是特殊符号类等等
3.重要级别： 测试用例的优先级 高/中等/低
4.操作步骤： 如何完成这个测试，比如 首先输入123，然后点击登录
5.期望结果： 预期得到什么结果
6.实际结果： 实际上得到了什么结果，当实际结果和预期结果不一致，可能有缺陷
我们用登录来简单看一个测试用例内容:

功能模块 检查项 操作步骤 重要级别 期望结果 实际结果 测试人
登录 界面提示 1.输入正确的用户名和密码，点击登录 高 登录进产品 xxx xxx

测试用例设计方法

场景设计法

在企业当中，几乎所有的产品软件都是用事件触发来控制业务流程的，事件触发时的情景形成场景，不同的顺序形成了事件流。
案例1：测试字母判定
要求： 第一列字符必须是A或B，第二列字符必须是一个数字，只有这样，允许修改文件
如果是第一列不符，给出L；如果是第二列不符合,给出M
基本事件流： A||B —-》数字—–》修改文件
接下来就分别针对基本事件流的每个部分进行考虑，可以得到5条测试用例
A—-》数字—-》修改文件
B—-》数字—-》修改文件
非A 非B —-》L
A—-》非数字—-》M
B—-》非数字—-》M
这就是场景设计法
案例2： QQ安装
基本事件流： 点击setup.exe进入安装程序 —-》选择安装路径—-》同意协议 —-》安装

  1. 点击setup.exe进入安装程序—-》选择默认路径 —-》 同意协议 —-》 安装
  2. 点击setup.exe进入安装程序—-》选择自定义路径—-》同意协议—-》安装
  3. 点击setup.exe进入安装程序—-》不填写路径
  4. 点击setup.exe进入安装程序—-》选择安装路径—》不同意协议
  5. 点击setup.exe进入安装程序—-》选择安装路劲—-》同意协议—-》取消安装
    当然，在整个事件流测试用例设计中，也需要和等价类划分、边界值结合起来设计。

    等价类划分

    我们把被测对象输入的数据叫做等价类，能符合被测对象要求的叫做有效等价类，不符合要求的叫做无效等价类。通过具体的例子来了解。

1.客户名字必须由汉字组成，那么汉字就是有效等价类，非汉字就是无效等价类
2.客户的名字长度必须是5-10位，那么 5-10位就是有效等价类，小于5或者大于10就是无效等价类
案例：邮箱注册
用户名为6-18个字符构成，包括字母、数字、下划线;
用户名以字母开头，字母或者数字结尾，不区分大小写
密码不能为空，确认密码需要和密码一致。

边界值

边界值常常与等价类划分联合使用，边界值考虑三个点，我们用6-18来说明
上点： 边界上的点
6和18就是输入域6-18边界上的点
离点： 离边界值最近的一个点
对于6-18来讲的话，如果是闭合区间，那么离点在外 5和19; 如果是开区间，那么离点在内 7和17
内点： 就是输入域中随意的值 比如 11

判定表

判定表非常适合那种有多个条件的场景，看个案例
案例1: 读书选择
疲倦是1 不疲倦是0 感兴趣 1 不感兴趣0 迷糊 1 不迷糊 0
1.觉得疲倦，但对书的内容感兴趣，同时书中的内容让你迷糊，回到本章重读
2.觉得疲倦，但对书中内容感兴趣，同时书中的内容不让你迷糊，继续读下去
3.不觉得疲倦并且对书的内容感兴趣，但是书中的内容让你迷糊的话，回到本章重读
4.觉得疲倦并且对书中的内容不感兴趣，同时书中的内容不让你迷糊，停止阅读，请休息
5.觉得疲倦并且对书中的内容不感兴趣，同时书中的内容让你迷糊，停止阅读，休息
6.不疲倦，对书中的内容感兴趣，书中内容不糊涂，继续阅读下去
7.不疲倦，不感兴趣，书中内容糊涂，跳到下一章阅读
8.不疲倦，不感兴趣，书中的内容不糊涂，跳到下一章阅读
我们看下这个需求，它就是由很多的条件组成，最后根据这些条件产生一个动作，这非常符合判定表的场景

1 2 3 4 5 6 7 8
条件 是否疲倦 1 1 0 1 1 0 0 0
是否感兴趣 1 1 1 0 0 1 0 0
是否糊涂 1 0 1 0 1 0 1 0
结果 本章重读 1 - 1 - - - - -
继续读 - 1 - - - 1 - -
停止阅读 - - - 1 1 - - -
跳到下一章 - - - - - - 1 1
有的书上会教判定表合并，我个人觉得不超过10条的，就不用合并了，容易漏掉用例，合并规则就是找相同的结果并且条件有且只有一个不同的。比如 7 8 结果都是1 条件里只有一个不同，那就合成一条就行。

因果图

案例1： 预售房预售资金监管账户网签号校验
对网签号格式进行验证，必须符合Y+7位数字格式，如Y2014678，如果符合要求，则可以验证成功，若第一列不是Y，则提示”网签格式错误”，如果后7位非数字，则提示”无此网签号”。
因果图需要理清输入和输出，输入和输入的逻辑关系
C1： 第一列是Y
C2： 其他7位是数字
E1： 网签号格式错误
E: 成功验证
E2： 无此网签号
所以可以得到 E=and(C1,C2) E1=not(C1) E2=not(C2)根据因果图设计判定表如下

1 2 3 4
原因(输入) 第一列是Y(C1) 0 0 1 1
7位数字(C2) 0 1 0 1
影响(输出) 成功验证(E) 0 1 0 1
网签格式错误(E1) 1 - - -
无此网签号(E2) - 1 - -
